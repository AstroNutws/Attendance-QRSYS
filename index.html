Security Architecture
Zero-Trust Design

Security Framework

Every layer of QrSys is engineered with data isolation and access scoping as first principles.

🛡️
Data Isolation
QrSys is middleware only. The development team holds zero direct ownership of your database at any point.
🔑
Client Control
Your organization retains full Supabase ownership — SQL execution and master keys stay exclusively with you.
👁️
Credential Integrity
Credentials are masked at UI level. Upcoming releases introduce one-way cryptographic hashing for all stored data.
⚙️
RBAC Enforcement
Strict role scoping throughout. Standard Admins handle operations; only Super Admins manage account creation.

Frequently Asked Questions
FAQ

Common Questions

What is the difference between Admin types?▼
Standard Admins handle day-to-day operations like monitoring and student lookup. Super Admins are system overseers with exclusive authority to create accounts, update credentials, and access the Super Admin panel.
What can the Super Admin do?▼
Super Admins can register new administrators, modify peer details (emails, usernames), reset passwords, and delegate Super Admin privileges to other accounts via the dedicated Super Admin Panel.
Who has direct access to the database?▼
Neither Standard nor Super Admins have direct database access. Ownership is strictly isolated to the Client via their private Supabase Dashboard — system management stays fully separated from data storage.
How are passwords and credentials secured?▼
Passwords are visually masked in the UI to prevent shoulder-surfing. Upcoming iterations will introduce server-side cryptographic hashing for enhanced backend security.
