Security Architecture
Zero-Trust Design

Security Framework

Every layer of QrSys is designed with data isolation and access control as first principles.

🛡️
Data Isolation
QrSys acts as a middleware layer. The dev team holds zero direct ownership of the database at any point.
🔑
Client Control
The organization retains full ownership of the Supabase project — SQL, master keys, and dashboard access.
👁️
Credential Integrity
Credentials are masked at UI level. Upcoming iterations will incorporate one-way cryptographic hashing.
⚙️
RBAC Enforcement
Strict role-based scoping. Standard admins handle operations; only Super Admins manage account creation.

FAQ & Support

What is the difference between Admin types? ▼
Standard Admins handle daily operations like monitoring and student lookup. Super Admins act as system overseers with exclusive authority to create admin accounts, update credentials, and access the Super Admin management panel.
What exclusive powers does the Super Admin have? ▼
Super Admins access the Super Admin Panel to register new administrators, modify peer details (emails, usernames), reset passwords, and toggle Super Admin status for others to delegate authority effectively.
Who has direct access to the database? ▼
Neither Standard nor Super Admins have direct database access. Ownership is strictly isolated to the Client (Database Owner) via their private Supabase Dashboard — ensuring system management remains separated from data storage for maximum security.
How are passwords and credentials protected? ▼
Passwords are visually masked in the application UI with circular symbols to prevent shoulder-surfing. Future updates will include server-side cryptographic hashing for enhanced backend security and complete one-way protection.
